## What is a system block?

A **system block** is a labelled chunk of context the API stitches together into the LLM's system prompt for each conversation. Examples:

- "We sell scientific lab equipment to UK universities."
- "Free shipping on orders over £50."
- "Top pages this month: …" (auto-generated by the Independent Analytics integration)
- "Always recommend the calculator demos before suggesting products."

The set of enabled blocks **is** the chatbot's personality + factual context. Today there's no way for the WP site admin to manage these blocks themselves; they're hand-edited on the API side. We need to give the admin CRUD over them, with a clear distinction between hand-written blocks and integration-generated blocks.

---

## Block data model

```json
{
  "id": "blk_2c4f7a9d",
  "title": "Store policies",
  "body": "We offer free shipping on UK orders over £50 …",
  "priority": 50,
  "enabled": true,
  "source": "manual",
  "tags": ["policy", "shipping"],
  "created_at": "2026-05-12T10:14:00Z",
  "updated_at": "2026-05-19T08:14:00Z"
}
```

Field notes:

- `priority` — integer; lower = earlier in the assembled prompt. Default 50.
- `source` — one of `manual`, `integration:woocommerce`, `integration:independent-analytics`, etc. The admin can only edit `body` / `title` on `manual` blocks; for integration blocks they can toggle `enabled` and adjust `priority`.
- `tags` — free-form, used in admin filtering. Not surfaced to the model.

Hard limits:
- `body`: 4000 chars per block. Pushed back via 422 on save.
- Total enabled blocks per chatbot: 50. Pushed back via 409 on enable.

## Architecture options

### Option A — Blocks live on the API (recommended)

The upstream API is the system of record. The plugin is a thin editor: it `GET`s the list, the admin tweaks, and the plugin `PATCH`es / `POST`s back.

Pros:
- Single source of truth across multiple WP installs that share an API instance.
- Plugin can be uninstalled without losing the curated prompt.
- The API can use the blocks immediately at chat time without any extra round-trip.

Cons:
- Plugin is offline-useless for block editing (no local fallback).
- Requires server-side HTTP calls from the plugin — a new direction (the plugin has been client-side-only until now).
- API auth must work in admin context (key in plugin settings, see below).

### Option B — Blocks live in WordPress, pushed to API

The plugin owns the blocks (custom post type or custom table) and syncs to the API on save and on schedule.

Pros:
- Block editing works even when the API is unreachable.
- Native WP editing UX (CPT lists, REST API, capabilities).

Cons:
- Two systems of truth — conflict resolution becomes a recurring source of bugs.
- Integration blocks would have to live on the API anyway (they're generated from WP data but combined with manual blocks at chat time, so storing only WP-side splits the data).
- Plugin uninstall = data loss unless we publish to the API on every save.

### Recommendation

**Option A.** The simplicity of one source of truth outweighs the offline-edit benefit. We mitigate the "offline-useless" downside with a clear error state in the admin UI ("Site Walker API unreachable — block editing temporarily disabled").

The rest of this doc assumes Option A.

## Upstream API routes (need to be built)

All require API key auth (admin-scoped — see auth section below).

```
GET    /admin/blocks                  - list, filterable by source / enabled / tag
POST   /admin/blocks                  - create a manual block
GET    /admin/blocks/{id}             - fetch one
PATCH  /admin/blocks/{id}             - edit
DELETE /admin/blocks/{id}             - delete (only manual blocks; integration blocks are gone when the integration retracts them)
POST   /admin/blocks/preview          - given a candidate set of blocks, return the assembled system prompt for visual inspection
PUT    /admin/blocks/integration/{slug}  - bulk-replace all blocks for a given integration source (atomic; used during sync)
```

### `PUT /admin/blocks/integration/{slug}` semantics

The plugin pushes the integration's full current block set in one call. The API:
1. Lists existing blocks where `source == "integration:{slug}"`.
2. Diffs incoming vs existing by `external_id` (stable per-integration identifier).
3. Inserts new, updates changed, deletes missing.
4. Preserves admin-set `enabled` / `priority` on updates.

Atomic — either the whole replace succeeds or none of it is applied.

### Why not `PUT /admin/blocks` with the full universe?

Because manual blocks live alongside integration blocks. A plugin sync should never touch manual blocks. Per-integration scoping keeps the contract clean.

## Plugin-side surface

### Settings tab

New "Context" tab on the admin page (alongside General, Appearance, and the planned Integrations tab). Sub-views:

- **Blocks list** — table of all blocks, with `source`, `enabled` toggle, priority spinner, edit / delete actions.
- **Edit block** — modal or sub-page. Title + body (textarea, char-counter). Tag input.
- **Preview** — button that calls `POST /admin/blocks/preview` with the current enabled set and renders the assembled prompt in a `<pre>` block. Helpful for sanity-checking ordering.
- **Refresh integration blocks** — button that triggers an immediate sync (otherwise sync runs on wp-cron).

### Server-side HTTP client

This is the first time the plugin talks to the upstream API server-side. New helper:

```php
namespace Site_Walker;

class Api_Client {
    public function get( string $path, array $query = [] ): array {}
    public function post( string $path, array $body ): array {}
    public function patch( string $path, array $body ): array {}
    public function delete( string $path ): array {}
}
```

- Uses `wp_remote_request()` underneath.
- Reads the API URL from `OPT_API_URL` (existing setting).
- Reads the admin API key from a new `OPT_ADMIN_API_KEY` setting.
- 5-second connect timeout, 15-second read timeout.
- Returns parsed JSON arrays, or throws a typed exception on transport / non-2xx.
- Logs failures via `error_log()` with the request path but never the body (avoid leaking block content into PHP logs).

### Caching

The admin block list is cached in a transient (`site_walker_wp_blocks_cache`, 5 min TTL) so navigating between admin tabs doesn't slam the API. Save / delete / sync busts the cache.

## Sync model (integration blocks → API)

```
wp-cron, daily          ┐
admin "Refresh now"     ┼─→  iterate active integrations
integration data event  ┘        │
                                 ▼
                       For each integration:
                         blocks = $integration->generate_blocks()
                                 ▼
                       PUT /admin/blocks/integration/{slug}  (atomic replace)
```

Integration data events worth wiring up:
- WC product save / delete → mark the WC integration's blocks dirty (refresh on next cron tick rather than immediately, to avoid storms on bulk imports).
- Analytics data refresh → re-generate the IA integration's blocks.

A "dirty" flag is a per-integration option (`site_walker_wp_int_{slug}_dirty = true`). Cron checks it; if set, runs sync for that integration only.

## Auth — admin key

The visitor-facing flow has its own key (M2 milestone). The admin direction wants a **separate** key:

- `OPT_VISITOR_API_KEY` — sent by the front-end JS. Lower-privilege, can mint sessions.
- `OPT_ADMIN_API_KEY` — sent by the plugin server-side for block CRUD and integration sync. Higher-privilege.

**Why two:** the visitor key is necessarily exposed in JS (and to anyone who views source). It must not be able to edit blocks.

If only one key is configured, integration sync and block editing degrade gracefully:
- List + preview disabled in the UI with a clear "Configure your admin key" prompt.
- Visitor flow continues to work.

## Permissions

- `current_user_can( 'manage_options' )` for all block CRUD in the admin UI (same as plugin settings today).
- All plugin-side AJAX/REST routes verify capability + a nonce.
- Capability hook `site_walker_can_manage_blocks` exposed for sites that want a finer-grained role.

## Open questions for review

1. **Multi-chatbot per API instance.** If one API instance serves multiple WP sites (each a separate chatbot under one account, post-v0.16), do blocks scope per-chatbot automatically (by chatbot slug derived from the account admin key + a per-chatbot identifier) or do we need an explicit `chatbot_slug` in the routes? Recommend the former — derive from auth + the targeted resource. Note: the upstream `/admin/chatbots/{slug}/blocks` surface already exists for filesystem-backed disk blocks; this plugin-side proposal is for the richer DB-backed catalogue, and may end up sitting on top of (or alongside) that route.
2. **Block versioning / history.** Do we want an audit trail (who changed what, when, undo)? Nice but not v1.
3. **Markdown in block bodies.** Allow / strip / require? Recommend: store raw text, no inline markdown. The API can normalise.
4. **Conflict resolution on concurrent edits.** Two admins editing the same block. Optimistic locking via `If-Match: <updated_at>` header. 412 on mismatch.
5. **What's the import / export story?** A "Download all blocks as JSON" affordance would make it easy for a merchant to back up their curated prompt. Probably v2.
6. **Preview rendering — verbatim or with placeholders?** The preview should show the prompt the model will actually see, including any `{visitor_context}` placeholders the API substitutes at chat time. Just literal substitution with sample values, no live model call.

## Implementation order (suggestion)

1. **Upstream API side first.** No point building plugin UI against routes that don't exist. Build `GET / POST / PATCH / DELETE /admin/blocks` and `POST /admin/blocks/preview`. Add admin-key auth.
2. **Plugin: `Api_Client` + admin key setting.** Smallest piece that proves the new server-side direction works.
3. **Plugin: "Context" tab read-only — list + preview.** No edit yet. Tests the read path.
4. **Plugin: manual block create / edit / delete.**
5. **Upstream API: `PUT /admin/blocks/integration/{slug}`.**
6. **Plugin: integration sync (cron + dirty flag) using WC integration as the first consumer.**
7. **Plugin: Independent Analytics integration as the second consumer.**

Steps 1–4 are independent of the integrations work and can ship while we're still designing those.
